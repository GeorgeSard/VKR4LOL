# Описание панелей дашборда «VKR — Flight Delay API мониторинг»

URL: http://localhost:3000/d/vkr-flight-delay (admin / admin)

Дашборд минимальный — 8 панелей, разбитых на 3 секции. Что хотел показать каждый блок и
зачем именно эти метрики важны для ML-сервиса.

---

## Секция 1. Общие показатели — текущее состояние API (3 stat-панели)

Вверху три карточки «здесь и сейчас». Их видно с одного взгляда — удобно во время демо
для защиты ВКР.

| Панель | Что показывает | Метрика | Норма |
|---|---|---|---|
| **Общий RPS** | Сколько HTTP-запросов в секунду обрабатывает FastAPI прямо сейчас. | `sum(rate(http_requests_total{job="fastapi"}[1m]))` | 0–10 фон, >10 нагрузка |
| **P95 latency** | 95% запросов завершаются быстрее этого значения. Это «длинный хвост»: даже если медиана быстрая, p95 покажет редкие медленные запросы. | `histogram_quantile(0.95, ...)` | <100ms — зелёный, <500ms — жёлтый |
| **Error rate** | Доля 4xx (ошибки клиента, например 422 валидация Pydantic) и 5xx (ошибки сервера) от общего числа запросов. | `(5xx + 4xx) / total` | <1% |

---

## Секция 2. FastAPI — трафик, латентность, ошибки (3 timeseries)

Главные графики проекта. Они отвечают на 3 вопроса в защите:
- «Какой эндпоинт нагружен?» — `/predict` должен быть основной нагрузкой
- «Быстро ли модель отвечает?» — p95 для `/predict` должен быть низким
- «Стабильно ли работает API?» — статусы должны быть только 2xx

| Панель | Что показывает | Зачем |
|---|---|---|
| **RPS по эндпоинтам** | Stacked area chart: запросы в секунду на каждый эндпоинт (`/predict`, `/health`, `/metrics`, `/experiments`, …) | Видно какой эндпоинт самый нагруженный. `/metrics` дёргает Prometheus каждые 15 сек — это фон, его всегда видно. |
| **p50/p95 latency по эндпоинтам** | Медианная и 95-перцентильная латентность отдельно для каждого эндпоинта | `/predict` должен быть быстрым (~5-50ms на CPU). Если p95 сильно больше p50 — производительность неравномерная. |
| **HTTP статус-коды** | Bar-chart с накоплением: 2xx (успех), 3xx, 4xx (ошибки клиента), 5xx (ошибки сервера) | В норме видно только 2xx. Появление 5xx → `docker compose logs -f fastapi`. |

---

## Секция 3. Ресурсы стека — память модели и CPU (2 timeseries)

Минимальный набор: видно что **модели реально загружены в RAM** и что **стек CPU-стабилен**.
Сети, дисков, дескрипторов — нет, для ML-проекта это не критично.

| Панель | Что показывает | Норма |
|---|---|---|
| **FastAPI: память процесса** | RSS (резидентная память — реально занятый RAM) и Virtual (адресное пространство). RSS — это что считает ОС за «память процесса». **В RSS включены загруженные ML-модели — поэтому это и есть доказательство что модели в памяти.** | RSS ~250-500MB после загрузки моделей |
| **CPU Docker и хоста** | CPU всех Docker-контейнеров суммарно и CPU всей Docker-VM. | <50% при средней нагрузке, скачки при `dvc repro` (тренер) |

⚠️ **Про macOS Docker Desktop:** на macOS cAdvisor не отдаёт метрики **по именам контейнеров** (это известное ограничение), поэтому CPU Docker — агрегат `id="/docker"` (все контейнеры разом). Для production-Linux это была бы детализация по каждому сервису.

## Что было удалено и почему

Из изначальных 15 панелей убрано 7:

| Удалено | Почему |
|---|---|
| Всего обработано запросов | Кумулятивный счётчик, для демо неинтересно — RPS показательнее |
| Размер запросов и ответов | Для ML-проекта не критично (тело `/predict` ≈ 1KB всегда) |
| FastAPI: CPU процесса | Дублирует CPU Docker и хоста |
| FastAPI: открытые дескрипторы | Для ML-сервиса нерелевантно (тут нет долгих TCP-сессий) |
| Память Docker (всего по cgroup) | Для ML важна память **именно FastAPI** (модели), а не общая |
| Сетевой трафик Docker | Для ML с small JSON-телами не показатель |
| Дисковый I/O Docker | Тренер пишет модели редко, FastAPI читает 1 раз при старте — графика не интересно |

## Как генерировать нагрузку для демонстрации

```bash
# burst 200 запросов на /predict
for i in {1..200}; do
  curl -s -X POST http://localhost:8000/predict \
    -H "Content-Type: application/json" \
    -d '{"scheduled_departure_local":"2025-11-29 08:35","airline_code":"FV","origin_airport":"TJM","destination_airport":"SVO","aircraft_type":"A321","distance_km":1769,"planned_duration_min":201,"weather_origin":"fog","weather_destination":"clear","temperature_origin_c":-3.9,"wind_speed_origin_mps":3.8,"precipitation_origin_mm":0.2,"visibility_origin_km":1.9,"airport_load_index":0.591,"airline_load_factor":0.787,"previous_flight_delay_min":38,"route_avg_delay_min":6.5,"aircraft_age_years":14,"technical_check_required":0,"crew_change_required":0}' \
    > /dev/null
done

# постоянная нагрузка (10 запросов/сек на 30 сек)
for i in {1..300}; do
  curl -s http://localhost:8000/health > /dev/null
  sleep 0.1
done
```

После этого все панели наполнятся данными. Refresh интервал дашборда — 10 секунд.

## Что делать, если на панели «No data»

1. **Проверь Prometheus targets:** http://localhost:9090/targets — все должны быть UP.
2. **Проверь, что FastAPI отдаёт метрики:** `curl http://localhost:8000/metrics` — должно прийти ~200 строк.
3. **Проверь cAdvisor:** `curl http://localhost:8080/metrics | grep container_memory_usage_bytes | head` — должны быть строки с `id="/docker"`.
4. **На macOS:** если запросов нет, метрики типа RPS будут 0 — это норма, не баг. Сгенерируй нагрузку (выше).
5. **Refresh:** в правом верхнем углу Grafana есть кнопка обновления, переключи на «Last 5 minutes» если данные старые.
