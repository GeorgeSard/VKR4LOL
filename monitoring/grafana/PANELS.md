# Описание панелей дашборда «VKR — Flight Delay API мониторинг»

URL: http://localhost:3000/d/vkr-flight-delay (admin / admin)

Дашборд разбит на 4 секции (rows). Каждая панель имеет тултип-описание прямо в Grafana — наведи на иконку «i» в правом верхнем углу панели.

---

## Секция 1. FastAPI — общие показатели (4 stat-панели)

| Панель | Что показывает | Метрика | Норма |
|---|---|---|---|
| **Общий RPS** | Сколько HTTP-запросов в секунду обрабатывает FastAPI прямо сейчас (среднее за минуту). Один из главных индикаторов нагрузки. | `sum(rate(http_requests_total{job="fastapi"}[1m]))` | 0–10 — фон, >10 — нагрузка |
| **Всего обработано запросов** | Кумулятивный счётчик: сколько запросов FastAPI обработал с момента старта. Сбрасывается при перезапуске контейнера. | `sum(http_requests_total{job="fastapi"})` | растёт монотонно |
| **P95 latency** | 95% запросов завершаются быстрее этого значения. p95 — это «длинный хвост»: даже если медиана быстрая, p95 может выявить редкие медленные запросы. | `histogram_quantile(0.95, ...)` | <100ms зелёный, <500ms жёлтый |
| **Error rate** | Доля 4xx (клиентские ошибки, например 422 валидация Pydantic) и 5xx (серверные ошибки) от общего числа запросов. | `5xx+4xx / total` | <1% |

## Секция 2. FastAPI — детали по эндпоинтам (4 timeseries)

| Панель | Что показывает | Зачем |
|---|---|---|
| **RPS по эндпоинтам** | Stacked area chart: сколько запросов в секунду на каждый эндпоинт (`/predict`, `/health`, `/metrics`, `/experiments`, и т.д.). | Видно, какой эндпоинт самый нагруженный. `/metrics` дёргает Prometheus каждые 15 сек — это фон. |
| **p50/p95 latency по эндпоинтам** | Медианная и 95-перцентильная латентность отдельно для каждого эндпоинта. | Если p95 сильно больше p50 — значит производительность неравномерная. `/predict` должен быть быстрым (~5-50ms на CPU). |
| **HTTP статус-коды** | Bar-chart с накоплением: сколько 2xx (успех), 3xx (редирект), 4xx (ошибки клиента), 5xx (ошибки сервера). | В норме видно только 2xx. Появление 5xx → смотри `docker compose logs -f fastapi`. |
| **Размер запросов и ответов (p95)** | Тело входящих JSON и исходящих ответов (95-перцентиль в байтах). | Тело `/predict` ≈ 1KB, ответ ≈ 150 байт. Резкий рост = аномалия. |

## Секция 3. FastAPI — процесс Python (3 timeseries)

| Панель | Что показывает | Норма |
|---|---|---|
| **Память процесса** | RSS (резидентная память — реально занятый RAM) и Virtual (адресное пространство). RSS — это что считает ОС за «память процесса». В RSS включены загруженные ML-модели. | RSS ~250-500MB, virtual ~2GB |
| **CPU процесса** | Доля CPU, потребляемая процессом FastAPI. 1.0 = одно ядро полностью занято. | <0.5 при средней нагрузке |
| **Открытые дескрипторы** | Сколько файловых дескрипторов открыто (HTTP-соединения, файлы, сокеты). | стабильно 10-50 |

Зачем эта секция: process_* метрики идут от `prometheus_client` напрямую из Python и работают на любой ОС. Они не зависят от cAdvisor.

## Секция 4. Контейнеры (cAdvisor — агрегаты)

⚠️ **Важно про macOS Docker Desktop:** на macOS cAdvisor видит контейнеры только агрегированно по cgroup-путям (`id="/docker"` — все Docker-контейнеры разом, `id="/"` — вся VM Docker Desktop). Имена индивидуальных контейнеров (`vkr-fastapi`, `vkr-mlflow` и т.д.) на macOS **недоступны** — это известное ограничение Docker Desktop, на Linux работает иначе.

Поэтому в этой секции — только агрегаты.

| Панель | Что показывает | Зачем |
|---|---|---|
| **Память Docker (всего по cgroup /docker)** | Суммарная память всех Docker-контейнеров: usage_bytes (cache + rss + swap) и rss (только активная память). | Увидеть общую нагрузку на RAM от стека. |
| **CPU Docker и хоста** | CPU всех Docker-контейнеров суммарно и CPU всей Docker-VM. | При запуске trainer (CPU-heavy) видно скачок. |
| **Сетевой трафик (Docker eth0)** | Входящий (RX) и исходящий (TX) сетевой трафик на главном интерфейсе. | Видно, как Prometheus тянет метрики, как FastAPI отдаёт ответы. |
| **Дисковый I/O Docker** | Чтения и записи на диск всеми Docker-контейнерами (rate байт/сек). | Тренер пишет модели и метрики, FastAPI читает pkl при загрузке. |

## Как генерировать нагрузку для демонстрации

```bash
# burst 200 запросов
for i in {1..200}; do
  curl -s -X POST http://localhost:8000/predict \
    -H "Content-Type: application/json" \
    -d '{"scheduled_departure_local":"2025-11-29 08:35","airline_code":"FV","origin_airport":"TJM","destination_airport":"SVO","aircraft_type":"A321","distance_km":1769,"planned_duration_min":201,"weather_origin":"fog","weather_destination":"clear","temperature_origin_c":-3.9,"wind_speed_origin_mps":3.8,"precipitation_origin_mm":0.2,"visibility_origin_km":1.9,"airport_load_index":0.591,"airline_load_factor":0.787,"previous_flight_delay_min":38,"route_avg_delay_min":6.5,"aircraft_age_years":14,"technical_check_required":0,"crew_change_required":0}' \
    > /dev/null
done

# постоянная нагрузка (10 запросов/сек на 30 сек)
for i in {1..300}; do
  curl -s http://localhost:8000/health > /dev/null
  sleep 0.1
done
```

После этого все панели наполнятся данными. Refresh интервал дашборда — 10 секунд.

## Что делать, если на панели «No data»

1. **Проверь Prometheus targets:** http://localhost:9090/targets — все должны быть UP.
2. **Проверь, что FastAPI отдаёт метрики:** `curl http://localhost:8000/metrics` — должно прийти ~200 строк.
3. **Проверь cAdvisor:** `curl http://localhost:8080/metrics | grep container_memory_usage_bytes | head` — должны быть строки с `id="/docker"`.
4. **На macOS:** если запросов nem то метрики типа RPS будут 0 — это норма, не баг. Сгенерируй нагрузку (выше).
5. **Refresh:** в правом верхнем углу Grafana есть кнопка обновления, переключи на «Last 5 minutes» если данные старые.
